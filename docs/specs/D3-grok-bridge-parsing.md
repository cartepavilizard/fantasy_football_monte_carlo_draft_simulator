> skepticism model. The cheap half (a Sonnet session) implements
> `backend/models/player_notes.py`, the prompt templates, endpoints,

## 1. The core design move: parse at prompt time, not paste time

The backend has **no LLM and no xAI API access — by design** (the
brainstorm’s explicit constraint). So free-text → structure cannot
rely on smart parsing. Instead, the **prompt we generate instructs
Grok to end its answer with a fenced, line-keyed block**, and the
parser is a deterministic extraction of that block. The unstructured
prose above the block is kept verbatim as the human-readable note;
the block is the machine-readable part. When the block is missing or
mangled (user trimmed it, Grok ignored the instruction), parsing
degrades to “raw text + user fills the fields in the UI” — never to
guessing.

### The instructed footer (contract between template and parser)

Every generated prompt ends with:

```
Finish your answer with exactly this block, filled in:

---GROK-NOTE---
PLAYER: <player name>
STATUS_SIGNAL: one of UPGRADE | DOWNGRADE | UNCHANGED | UNCLEAR
SUMMARY: <one factual sentence, max 30 words>
SOURCES: <who said it and where, one per line, each with a date>
NEWEST_SOURCE: <date of the most recent source, YYYY-MM-DD>
CONFIDENCE: one of REPORTED | RUMORED | SPECULATION
---END-GROK-NOTE---
```

`STATUS_SIGNAL` is about the player’s availability/role trajectory in
the cited window. `CONFIDENCE` is Grok’s own sourcing claim —
REPORTED (named reporter, dated), RUMORED (aggregation/unsourced),
SPECULATION (Grok reasoning, no source).

## 2. Prompt templates (cheap half builds these; shapes are normative)

Generated by `GET /inseason/grok_prompt?player=<name>&kind=<kind>`
(pure string assembly from cached data — no fetch):

1. **`beat_check`** (the brainstorm’s example):
   “What has {beat_writer} ({outlet}) said about {player} in the last
   48 hours? Include exact quotes and dates. If they have said nothing
   in that window, say so explicitly — do not substitute older or
   national reporting.” + footer.
2. **`injury_timeline`**: “What is the most recent reporting on
   {player}’s {injury} recovery timeline? Cite each source with its
   date; distinguish team statements from reporter speculation.” +
   footer.
3. **`usage_context`** (pairs with a C4 alert): “Reporting from the
   last week on {player}’s role change ({context}) — is this
   coach-confirmed or observed only?” + footer.

Each template names the 48-hour/most-recent window **and** demands
dates — the parser’s staleness logic (§4) depends on Grok having been
told to date things.

## 3. Storage model

```python
class PlayerNote(Model):
    model_config = {"collection": "player_notes"}
    season: int
    week: int
    player_name: str
    nfl_team: Optional[str] = None
    kind: str                       # beat_check | injury_timeline | usage_context
    prompt_text: str                # exactly what we generated
    raw_text: str                   # exactly what the user pasted
    summary: Optional[str] = None           # from the block, or user-entered
    status_signal: Optional[str] = None     # upgrade|downgrade|unchanged|unclear
    grok_confidence: Optional[str] = None   # reported|rumored|speculation
    sources: List[str] = []
    newest_source_date: Optional[datetime.date] = None
    parsed_block: bool = False      # did the footer parse?
    stale_risk: bool = True         # see §4 — defaults to skeptical
    conflicts: List[str] = []       # vs official signals, see §4
    verified: bool = False          # ALWAYS False; no code path sets True
    created_at: datetime.datetime
```

`source`-style provenance is the model’s whole identity: `kind`,
`prompt_text`, `raw_text` make every note auditable back to what was
asked and what came back.

## 4. Skepticism handling — three independent checks

Grok output is unverified secondhand information that may be stale or
hallucinated. Three mechanical defenses, all visible in the UI:

1. **Staleness**: `stale_risk = newest_source_date is None or
   (now.date() − newest_source_date).days * 24 > GROK_STALE_HOURS
   (default 72)`. Undated claims are stale by definition — the
   template demanded dates; their absence is itself information.
   Note the field defaults to `True`: a note is skeptical until the
   parse proves freshness.
2. **Conflict with official signals** — computed at save time from
   already-synced data (Mongo only):
   - `status_signal == "upgrade"` while the player’s current ESPN
     roster status / D2 designation is `out`/`injury_reserve`
     → `conflicts += ["claims upgrade but official status is out (as of <sync time>)"]`.
   - `status_signal == "downgrade"` while official status is
     `active` and (when D2 is live) the newest same-week
     `PracticeReport.participation == "full"`
     → conflict recorded symmetrically. (This direction is expected
     sometimes — beat writers lead official reports; the conflict
     string says “ahead of official reports OR wrong”, and the UI
     shows it as a caution, not an error.)
   - `grok_confidence == "speculation"` → `conflicts +=
     ["Grok labeled this speculation — no source"]`.
3. **Quarantine from automation** (the load-bearing one):
   `verified: false` forever means **no automated consumer reads
   PlayerNote**. Not E1’s availability curves, not E4’s triggers, not
   C1’s warnings. The note renders on the player’s card/panel for a
   human, who may then act — e.g. by passing an explicit
   `availability_overrides` to E1’s evaluate. The human is the
   verification step; the bridge is a research assistant, not a
   sensor. Enforced by a test that greps the import graph: no module
   outside `player_notes.py`/its API imports `PlayerNote`.

## 5. Parser specification

```python
def parse_grok_paste(raw_text: str) -> dict   # pure function
```

- Locate the **last** `---GROK-NOTE--- … ---END-GROK-NOTE---` span
  (last: Grok sometimes echoes the instruction before answering).
  Tolerate leading whitespace, `>` quoting, and code fences around the
  markers; match markers case-insensitively.
- Within the span, parse `KEY: value` lines for the six keys.
  Multi-line `SOURCES` collects until the next known key or end.
  Unknown keys are ignored. Enum fields (`STATUS_SIGNAL`,
  `CONFIDENCE`) lowercase and validate; invalid → `None` (not an
  error).
- `NEWEST_SOURCE` parses `YYYY-MM-DD` (also tolerate `MM/DD/YYYY` and
  `Month D, YYYY`); unparseable → `None` → stale by rule §4.1.
- No block found → `{parsed_block: False}` with every structured field
  `None`; `raw_text` is still preserved and the UI prompts the user to
  set `status_signal`/`summary` manually before saving.
- The parser must never raise on any input (property test: arbitrary
  text → dict).

## 6. Endpoints (in `inseason_api.py`, standard patterns)

- `GET /inseason/grok_prompt?player=&kind=&season=` — template
  assembly; joins D1’s writer directory by the player’s `nfl_team`
  (from any current roster/FA row); errors 404 on unknown player,
  degrades gracefully on unknown writer (team-level phrasing).
- `POST /inseason/player_note/parse` — body `{raw_text}` → parse
  preview + computed `stale_risk`/`conflicts` **without saving**
  (the UI’s confirm screen).
- `POST /inseason/player_note` — body `{player_name, kind,
  prompt_text, raw_text, summary?, status_signal?, ...}` — re-runs
  parse + skepticism server-side (never trust the preview round-trip),
  saves, returns the stored note.
- `GET /inseason/player_notes?player=&week=&season=` — newest first.
- `DELETE /inseason/player_note/{id}` — notes are user data; full
  delete, no soft-delete ceremony.

All Mongo-only; add the GETs to both cached-only enforcement tests.
The POSTs write to Mongo but fetch nothing — they belong in
`inseason_api.py` (the cached-only rule bars external calls, not
writes; same reasoning as the handcuff CRUD).

Config: `GROK_STALE_HOURS=72`.

## 7. Edge cases (all must be tested)

- Paste contains **two** blocks (echoed instruction + real answer) →
  last block wins.
- Block present but `STATUS_SIGNAL: probably fine` → invalid enum →
  `None`; UI requires a manual choice before save.
- `SOURCES` says “no recent comments found” with
  `STATUS_SIGNAL: UNCLEAR` → perfectly valid, stores as an
  informative negative (the beat writer’s silence is a data point);
  stale rule applies off `NEWEST_SOURCE`.
- Player name in block ≠ requested player → save under the
  **requested** player, append conflict
  `"Grok answered about <other name>"` — likely hallucination, highest
  skepticism.
- Paste is empty/whitespace → 422 on save; parse preview returns the
  no-block shape.
- Same player, multiple notes per week → all kept, newest first; no
  dedupe (research accumulates).
- Emoji/markdown/RTL text in `raw_text` → stored verbatim (test with
  one gnarly fixture).

## 8. Worked example

User hits a C4 alert on WR “U”, requests
`GET /inseason/grok_prompt?player=U&kind=usage_context`, runs it in
their free xAI account, pastes back an answer whose footer reads:

```
---GROK-NOTE---
PLAYER: U
STATUS_SIGNAL: UPGRADE
SUMMARY: Beat writer reports U ran with the first team all week after the WR2's benching.
SOURCES: J. Reporter (Team Blog), 2026-10-07; coach presser, 2026-10-06
NEWEST_SOURCE: 2026-10-07
CONFIDENCE: REPORTED
---END-GROK-NOTE---
```

Saved on 2026-10-08: `parsed_block=True`, `stale_risk=False` (1 day <
72h), official status `active` → no conflicts, `verified=False`.
The note renders on U’s player card with a “manual Grok research —
unverified” badge; when the user later grades a trade for U they may
choose to trust it — the model never does it for them.

Counter-case: same paste but `NEWEST_SOURCE` missing → `stale_risk =
True`, UI badge “undated sources — treat as stale”.

## 9. What the implementing session (Sonnet) must NOT do

- **No LLM/xAI/API calls anywhere** — string templates out, regex
  parsing in. This is the feature’s identity, not a limitation to fix.
- **No smart/fuzzy free-text parsing** beyond the specified block
  extraction and date tolerance — the prompt footer is the parser’s
  contract; improve the prompt, not the parser.
- **No automated consumption of PlayerNote** (§4.3’s quarantine, with
  its import-graph test).
- **No auto-setting `verified`** and no “confidence score” synthesis —
  the three skepticism fields are shown raw.
- **No storing xAI credentials or automating the browser** — the human
  copy/paste IS the design.
- Tests: parser property + fixture suite (every §7 case), skepticism
  matrix (stale × conflict × confidence), endpoint round-trips,
  quarantine import test.
