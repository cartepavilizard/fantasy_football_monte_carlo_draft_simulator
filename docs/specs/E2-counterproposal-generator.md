# E2 — Counterproposal Generator (spec)

> **Status:** Design locked (2026-07-11, Fable design pass). Implement
> verbatim in `backend/models/counterproposals.py`. Depends on E1
> (`models/trade_valuation.py`) being landed — every quantity here is
> in E1’s units and every evaluation goes through E1’s pure functions
> on a single shared `ValuationContext`.

## 1. Problem statement

Input: a proposed trade between team A and team B that E1 graded
outside its fairness bound (or that the user simply wants alternatives
for). Output: **1–3 counterproposals** that (a) keep the original
deal’s intent, (b) close the market gap into E1’s fair bound, and
(c) don’t wreck either roster’s fit. The generator is a search over
modifications of the proposal — not an open-ended “find any trade
between these teams” engine (that’s E4’s scanner, and even E4 only
flags windows).

Convention used throughout: the **disadvantaged** side is the one the
E1 verdict says the deal favors *against* — if `verdict = favors_b`,
team A is disadvantaged (sending more than it gets).

## 2. Search space — three move types, anchored

The original proposal is `(S_A, S_B)`. Counters are generated by
applying exactly **one** move to it (single-move counters are what a
human trade partner recognizes as “the same deal, tweaked”; compound
rewrites read as a new deal and belong to the human):

1. **ADD** — the advantaged side adds one player from its roster to its
   outgoing set (they’re getting the better of it; they sweeten).
2. **REMOVE** — the disadvantaged side removes one player from its
   outgoing set (only when it sends ≥ 2 players).
3. **SWAP** — one player in either outgoing set is replaced by a
   different player from the same roster.

**Anchor rule:** the most valuable player (by E1 `player_value`)
*received by the disadvantaged side* is the deal’s reason for existing
and is never removed or swapped out. Without an anchor, the “best”
counter is always “cancel the trade”, which is useless output.

**Resulting set-size bound:** each side sends 1–3 players after the
move (`MAX_SIDE_PLAYERS = 3`). Bigger packages are noise — ESPN trades
above 3-for-3 essentially never execute.

## 3. Pruning — how the search stays small

Naively: ~16 addable/swappable players per move slot → a few hundred
raw candidates per proposal. The expensive step is E1’s `fit_delta`
(2 × H DP runs per counter), so the funnel is:

**Stage 0 — build once:** one `ValuationContext` for the league; E1
market values for every player on both rosters are then O(1) lookups.

**Stage 1 — candidate filter (cheap, market values only):**
- Compute the original `market_gap` and the direction to close.
- ADD candidates: players on the advantaged roster with
  `player_value <= |market_gap| * (1 + GAP_SLACK)` — adding someone
  worth far more than the gap overshoots into unfairness the other way.
  Also `player_value >= |market_gap| * GAP_MIN_FRACTION` — adding a
  0-point body closes nothing and insults the partner.
- REMOVE candidates: non-anchor players in the disadvantaged side’s
  outgoing set.
- SWAP candidates: for each non-anchor outgoing player `q`, replacement
  players `r` on the same roster with
  `|value(r) − value(q)| ∈ [GAP_MIN_FRACTION, 1 + GAP_SLACK] × |market_gap|`
  and the sign that closes the gap.
- `GAP_SLACK = 0.5`, `GAP_MIN_FRACTION = 0.25` (env-tunable).
- **Untouchables filter:** never add/swap-in a player the owning side
  clearly shouldn’t move — anyone whose removal drops that roster’s
  weekly starting lineup by more than `SURPLUS_COST_CEILING = 1.5`
  pts/week (computed once per roster player in Stage 0 via a single
  team DP with and without them, at week `w0` only — one week as a
  proxy, not the full horizon, precisely to keep Stage 1 cheap).

**Stage 2 — rank by residual gap:** for each surviving candidate
counter, the new market gap is pure arithmetic on cached values
(no DP). Keep the `MAX_FINALISTS = 12` counters with the smallest
`|new_market_gap|`, requiring `|new_market_gap| <= fair_bound`
(recomputed for the counter’s own totals, per E1 §4.3).

**Stage 3 — full E1 evaluation (expensive, ≤ 12 times):** run
`evaluate_trade` on each finalist. Discard any counter where either
side’s `fit_delta < FIT_FLOOR = −2.0` ROS points (a counter that
actively hurts a roster won’t be accepted, fairness or not; small
negatives survive because need asymmetry is the point of trading).

**Stage 4 — select and diversify:** sort survivors by
`min(fit_delta_a, fit_delta_b)` descending (maximize the worse side’s
outcome — the most *acceptable* counter, not the most extractive), tie
break on `|market_gap|` ascending, then on a deterministic key
(move type ADD < REMOVE < SWAP, then player ids) so identical inputs
always produce identical output. Return the top `MAX_COUNTERS = 3`,
skipping any counter whose player multiset duplicates an
already-selected one.

Worst case cost: 2 roster-scans of one-week DPs (Stage 0) + 12 full
evaluations (Stage 3). Well under a second; no caching layer needed.

Returns:

```json
{
  "original": <E1 evaluate_trade dict>,
  "counters": [
    {
      "move": {"type": "add", "team": "b", "player_id": 4046, "player_name": "..."},
      "sends_a": [...player ids...], "sends_b": [...],
      "evaluation": <E1 evaluate_trade dict>,
      "rationale": "Adding <name> (14.2 ROS pts) closes the 18.4-point gap to 4.2 — inside the fair range — and still upgrades their RB2."
    }
  ],
  "note": null | "original trade is already fair — counters below are strict upgrades" | "no fair counter exists within one move of this proposal"
}
```

- When the original is already fair, still run the search (the user
  asked for options) but say so in `note`.
- When Stage 3 leaves nothing, return `counters: []` with the
  no-fair-counter note — an honest empty answer, never a stretched one.
- `rationale` strings follow C8 framing: quote ROS/per-week numbers and
  the positional need, never last week’s box score.

Endpoint (in `inseason_api.py`, standard envelope, added to both
cached-only enforcement tests):

- `POST /inseason/league/{espn_league_id}/trade/counters` — same body
  as E1’s evaluate plus nothing new; internally: `build_context` →
  `evaluate_trade` → `generate_counters`.

Config: `MAX_SIDE_PLAYERS=3`, `GAP_SLACK=0.5`, `GAP_MIN_FRACTION=0.25`,
`SURPLUS_COST_CEILING=1.5`, `MAX_FINALISTS=12`, `FIT_FLOOR=-2.0`,
`MAX_COUNTERS=3` in `models/config.py`.

## 5. Edge cases (all must be tested)

- **1-for-1 lopsided trade**: REMOVE is unavailable (would empty a
  side); ADD and SWAP still apply. A 1-for-1 where the disadvantaged
  side sends the anchor’s counterweight often only counters via ADD →
  becomes 2-for-1; roster-size note applies (below).
- **Unequal player counts** (2-for-1 etc.): legal; the evaluation dict
  gains `"roster_size_note": "team A ends this trade 1 player over/under — a drop/add is required on execution"`. Informational only.
- **Anchor ambiguity** (two received players tied in value): higher
  `player_value` wins; tie → lower player_id. Deterministic.
- **Both fit deltas negative in the original**: still search — a
  counter can fix fit for one side; if nothing clears `FIT_FLOOR`, the
  empty-with-note answer is correct.
- **Advantaged side has no addable surplus** (every candidate is
  untouchable or outside gap bounds): ADD contributes nothing; SWAP may
  still work.
- **Gap enormous** (e.g. superstar for a kicker): Stage 1 bounds mean
  no single move closes it → empty counters + note. Correct: that
  proposal needs a rewrite, not a tweak.

## 6. Worked example

Original (from E1’s worked example, extended): A sends WR X
(value 29.9) + WR Z (8.1) = 38.0; B sends RB Q (18.3). Gap = +19.7,
bound = max(10, 0.15×38.0) = 10 → `favors_b`, A disadvantaged.
Anchor: RB Q (the only player A receives).

- ADD candidates from B’s roster: players valued in
  [0.25×19.7, 1.5×19.7] = [4.9, 29.6] that aren’t untouchable →
  say TE R (12.4) and WR S (22.0).
  - +TE R → new gap 19.7−12.4 = 7.3 ✓ inside bound.
  - +WR S → new gap −2.3 ✓.
- REMOVE from A’s outgoing: WR Z (8.1) → new gap 11.6 ✗ (> 10, close
  but out; discarded at Stage 2).
- SWAP A’s WR Z (8.1) for A’s bench TE (2.0): moves gap the wrong way ✗.

Stage 3 evaluates the two ADD finalists; suppose
`{+TE R: fit_a +6.2, fit_b +2.9}` and `{+WR S: fit_a +9.0, fit_b −3.1}`.
WR S fails `FIT_FLOOR` for B → discarded. Output: one counter
(+TE R), rationale quoting the 19.7 → 7.3 gap close and B’s +2.9 fit.

## 7. What the implementing session must NOT do

- **No compound moves** (two changes per counter) and no counter
  generation from scratch — one move, anchored, or nothing.
- **No full-horizon DP in Stages 0–2.** The one-week surplus proxy and
  market-value arithmetic are the point; the full evaluation budget is
  Stage 3’s ≤ 12 calls.
- **No randomness.** The tie-break chain must make output a pure
  function of the synced data.
- **No modifications to E1.** If a needed quantity is missing from
  `ValuationContext`, add a *reader* here, or flag the spec defect in
  the task row — don’t reshape E1’s API from a consumer.
- **No auto-sending or persisting counters** — this is a read that
  computes; E7 handles messaging, the human handles sending.
- Tests: pure-function tests with hand-built contexts covering every
  edge case above, plus one endpoint test; assert determinism by
  running the search twice.
