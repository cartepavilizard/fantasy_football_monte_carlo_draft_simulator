# Hawk UI design-sync notes

## Gotchas from the first build

- **`npm install` inside this package creates duplicate deps that break the
  APP's typecheck.** `frontend/packages/hawk-ui` isn't an npm workspace
  member of `frontend/` — it has its own `package.json`/`node_modules`.
  Because the app's root `tsconfig.json` `include` glob (`**/*.tsx`) also
  covers this package's `src/`, TWO copies of `@types/react` (or any dep
  with type-bearing exports, e.g. `react-icons`) in the resolution graph
  cause cascading `TS2786`/`ReactNode not assignable` errors across
  UNRELATED app files. After any `npm install` here, diff this package's
  `node_modules/{react,react-dom,@types/react,@types/react-dom,react-icons}`
  against `frontend/node_modules`'s versions and `rm -rf` any that differ —
  resolution falls through to the root copy via normal Node directory-walk,
  no reinstall needed. `clsx` and the two `@nextui-org/*` deps matched
  versions on the first install and were left in place.
- **`typeof SomeReactIconComponent` fails tsup's stricter `.d.ts` builder**
  even though the app's own `tsc --noEmit` accepts it. Use react-icons'
  own `IconType` (`import type { IconType } from "react-icons"`) for any
  field/prop that stores an icon component reference to render later
  (the `tagMeta` pattern in `draft-tag-badge.tsx`) — `IconType` is the
  JSX-safe alias react-icons ships for exactly this.
- **A stale `tsconfig.tsbuildinfo` can show an error that's already fixed.**
  The app's `tsconfig.json` has `incremental: true`. If a fix doesn't
  clear an error on the next `tsc --noEmit`, delete
  `frontend/tsconfig.tsbuildinfo` and `frontend/.next/cache/.tsbuildinfo`
  and re-run before assuming the fix didn't take.

## CSS: `cssEntry` is a generated file, not a straight pointer

`cfg.cssEntry` (`tokens/hawk-ui-styles.css`) is NOT auto-regenerated by
`npm run build` or the converter — it's a manually-produced combination
of two things, regenerate both on any re-sync where components' visual
styling or classnames changed:

1. **Tokens + base rules**: copied verbatim from
   `frontend/styles/globals.css` starting at the
   `/* ============ HAWK MODE TOKENS ============ */` comment through
   EOF (tokens, base html/body rules, keyframes, `.hawk-scroll`,
   `.trade-room-body`). `cssEntry` reaching OUTSIDE the package directory
   was tried first and silently skipped by the converter (`! cssEntry: …
   resolves outside the package — skipped`) — it must be copied in, not
   referenced.
2. **Compiled Tailwind utilities**: this package's components use plain
   Tailwind utility classnames (`flex`, `gap-2`, etc.) alongside
   `var(--token)` inline styles. `tailwind.config.js` in this package
   root is a deliberately SELF-CONTAINED CommonJS config (content scoped
   to `./src/**/*`, with the app's `theme.extend` + `nextui()` plugin
   block copied in verbatim) — NOT `presets: [require("../../tailwind.config.js")]`,
   because the app's config mixes a top-level ESM `import` with
   `module.exports`, which the bare `tailwindcss` CLI's plain `require()`
   can't load (Next.js's own build pipeline transpiles it; the CLI
   doesn't). Regenerate via:
   ```
   npx tailwindcss -c tailwind.config.js -i <tmp @tailwind-directives file> -o <tmp output>
   ```
   then concatenate: compiled Tailwind output first, tokens/base second
   (matches the app's own `globals.css` cascade order), into
   `tokens/hawk-ui-styles.css`.

**Drift risk**: no build-time link enforces that this package's
`tailwind.config.js` theme block or the copied token file stay in sync
with `frontend/tailwind.config.js` / `frontend/styles/globals.css`. If
either changes in the app, re-copy/re-generate before the next sync.

## Fonts

Anton / Barlow Condensed / Barlow are loaded via `next/font/google` in the
app (`frontend/config/fonts.ts`) — no static font files are committed
there. For this sync, real woff2 files were fetched directly from Google
Fonts' CSS2 API (weights matching `config/fonts.ts` exactly: Anton 400;
Barlow Condensed 500/600/700; Barlow 400/500/600/700) into
`fonts/fonts.css` + sibling `.woff2` files, wired via `cfg.extraFonts`.
**Fetching Google's CSS2 endpoint needs a modern browser User-Agent** — a
generic `Mozilla/5.0` UA gets served `.ttf` instead of `.woff2`.

## Known render warns (checked against on every re-sync)

- `[FONT_MISSING] "Impact"` — expected, not a gap. "Impact" is only the
  CSS fallback in `--font-display: "Anton", Impact, sans-serif` (a system
  font, never meant to ship as a webfont); Anton itself IS shipped.
- `[RENDER_THIN]` on `CornerBadge`, `EmptyStateHawk`, `HeroMascot`,
  `VictoryBadge` — pure decorative SVG mascot art with no text content by
  design; the validator's "no text" heuristic doesn't distinguish that
  from a genuinely broken render. Visually confirmed correct in the
  contact sheets.
- `[RENDER_THIN]` on `NotificationsPanel` — floor-card default renders
  just the bell icon with no notifications data (no dropdown open);
  correct behavior for that state, not broken.
- `tokens: 204 defined, 72 referenced (2 missing, below threshold)` —
  non-blocking per the validator's own threshold; not chased further.

## Re-sync risks

- The fetched font files are a point-in-time snapshot (Google Fonts CSS2
  URLs can rotate). If a re-sync's render check flags missing/broken
  fonts, re-fetch `fonts/fonts.css` rather than assuming the existing
  files are still valid.
- See "CSS" section above — `tokens/hawk-ui-styles.css` needs manual
  regeneration, it does not auto-update from the app's source files.
